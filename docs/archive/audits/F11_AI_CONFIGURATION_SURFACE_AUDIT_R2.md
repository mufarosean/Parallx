## Dead-Code Deletion Safety Analysis

### C4 (Persona) — Can safely delete?
- **Yes.** `persona.name`/`description`/`avatarEmoji` are never read outside `aiSettings/` internal plumbing (UI section, preset switcher display, legacy mapping). No OpenClaw consumer, no chat consumer, no service consumer.
- **Delete:** `personaSection.ts`, `IUnifiedPersonaConfig`, `persona` key in `IUnifiedAIConfig` + `DEFAULT_UNIFIED_CONFIG` + preset definitions.
- **Risk:** Persona switcher display in preset dropdown shows name — would need alternate display text. Low complexity.

### C10 (AISettingsService class) — Can safely delete?
- **Yes.** `new AISettingsService` has zero results. `UnifiedAIConfigService` implements `IAISettingsService` directly via its legacy compat methods. The standalone class is dead code.
- **Delete:** `AISettingsService` class from `aiSettingsService.ts`. Keep exports that other files import (the interface is in `aiSettingsTypes.ts`).
- **Risk:** None — verified no instantiation.

### C11/C13 (Dead UI knobs) — Can safely delete?
- **Partially.** The suggestion controls, persona controls, responseLength, and 4 agent fields can be removed from UI. The underlying config types can shrink.
- **Risk:** If `generateChatSystemPrompt` remains, it reads tone/focusDomain — but its output (`chat.systemPrompt`) is only used by the Preview Test button. Removing the generator would break preview, which could be acceptable.

### C14 (Preset system) — Can safely delete?
- **No.** The preset mechanism is sound. Only the built-in preset *content* needs updating to vary consumed fields. The mechanism (create/switch/merge) should be retained.
